Game Engine

How the crash/aviator game loop works — from round lifecycle to provably-fair crash points.

Overview

The game engine lives in App\Services\GameEngine and is driven by the php artisan game:start Artisan command (App\Console\Commands\StartGameLoop). It continuously cycles through rounds, broadcasting every state change over WebSockets so that all connected clients stay perfectly synchronized.

Round Lifecycle

Each round progresses through four phases:

Phase Duration What Happens
WAITING Configurable (default 3 s) Countdown between rounds. Broadcasts GameCountdown event with seconds remaining.
BETTING Configurable (default 10 s) Players can place bets. Broadcasts GameBettingStarted. Bets arrive via the API and are broadcast as BetPlaced.
RUNNING Until crash point reached Multiplier grows exponentially. Broadcasts GameStarted once, then GameMultiplierUpdated every tick (~100 ms). Players can cash out; BetCashedOut is broadcast on success.
CRASHED Instant Round ends. Broadcasts GameCrashed with the final crash point. Unsettled bets are marked as lost.

Multiplier Formula

The multiplier follows an exponential curve:

multiplier = e^(growthRate × elapsedSeconds)

Where growthRate defaults to 0.06 (configurable via GAME_GROWTH_RATE). Example values:

Elapsed Time Multiplier
0 s 1.00×
5 s 1.35×
10 s 1.82×
20 s 3.32×
30 s 6.05×
40 s 11.02×
50 s 20.09×
60 s 36.60×

Crash Point Generation

Crash points are generated using a provably-fair algorithm handled by App\Services\ProvablyFairService:

  1. A server seed is generated and its SHA-256 hash is published before the round.
  2. A combined string "{client_seed}-{nonce}" is HMAC-SHA256 hashed with the server seed.
  3. The first 8 hex characters are converted to an integer h.
  4. If h % 33 === 0 the round is an instant crash at 1.00× (≈3% chance).
  5. Otherwise: crash_point = floor((100 × e) / (e − h)) / 100, where e = 2^32.
  6. A 3% house edge is applied: crash_point = max(1, crash_point × 0.97).

Players can verify any round after it completes using the Provably Fair verification page.

Starting the Game Loop

# Run continuously (production)
php artisan game:start

# Run a single round (testing)
php artisan game:start --once

The command handles SIGTERM / SIGINT gracefully — it finishes the current round before shutting down. In production, run this under Supervisor (see Installation).

Important: Only one instance of game:start should run at a time. Running multiple instances will create duplicate rounds.

Broadcasting

All game events are broadcast on the public game channel:

Event Payload Frequency
GameCountdown secondsLeft Every second during waiting phase
GameBettingStarted round id, hash, betting duration Once per round
GameStarted round id Once per round
GameMultiplierUpdated multiplier, elapsed seconds Every ~100 ms
GameCrashed crash point, round data Once per round
BetPlaced bet data (user, amount, slot) When a player bets
BetCashedOut bet data, multiplier, payout When a player cashes out

Auto-Cashout Processing

Auto-cashout is processed server-side during the running phase. On each tick the engine iterates all active bets that have an auto_cashout_at value. If the current multiplier meets or exceeds the target, the bet is automatically cashed out and a BetCashedOut event is broadcast.

This ensures fairness — auto-cashouts execute at the exact server multiplier, not the client's rendered value.

Canvas Rendering

The frontend renders the multiplier curve on an HTML5 Canvas element via GameCanvas.vue. A requestAnimationFrame loop draws:

  • The exponential curve path from (0, 1×) to the current multiplier
  • A flying object emoji (default 🚀) at the tip of the curve
  • The current multiplier value as large centered text
  • Grid lines and axis labels

The canvas automatically adapts to the container size and supports both light and dark themes.
